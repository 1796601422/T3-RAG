description: 当用户希望撰写 PRD、创建产品需求文档或规划新功能时使用。通过需求访谈、必要的代码库/知识库探索、模块设计和测试决策，逐步形成 PRD。
---

当用户希望创建 PRD 时，按以下流程协助用户。你可以根据场景跳过不必要的步骤。

1. 先请用户详细描述想解决的问题、目标用户、业务背景、已有想法和可能的解决方案。

2. 如果当前任务涉及已有系统、已有代码或历史知识库，先探索现状，验证用户的假设，理解当前系统能力、约束和已有类似方案。

3. 通过访谈逐步澄清需求。围绕目标、用户、场景、边界、流程、规则、异常、权限、配置、数据、验收标准和上线策略持续追问，直到双方对问题和方案形成一致理解。

4. 梳理需要新增或修改的主要模块。优先寻找可以沉淀为“深模块”的能力。

深模块指的是：用简单、稳定、可测试的接口封装较多功能和复杂度，接口不频繁变化，内部实现可以独立演进。

5. 和用户确认模块划分是否符合预期，并确认哪些模块需要重点测试。

6. 当需求和方案足够清晰后，使用下面模板撰写 PRD。当前产品只需要在对话中输出 PRD，不需要自动提交 GitHub issue。

<prd-template>

## 问题陈述

从用户视角描述当前遇到的问题、痛点、机会或业务诉求。

## 解决方案

从用户视角描述拟解决方案，包括核心思路、预期体验和业务价值。

## 用户故事

使用编号列表列出尽可能完整的用户故事。每条用户故事使用以下格式：

1. 作为一个<角色>，我希望<能力或功能>，以便<收益或目的>。

<user-story-example>
1. 作为一个移动银行用户，我希望能查看账户余额，以便更好地判断自己的消费决策。
</user-story-example>

用户故事应尽量覆盖该功能涉及的主要角色、主流程、异常流程和边界场景。

## 实现决策

记录已经达成一致的实现决策，可以包括：

- 需要新增或修改的模块
- 模块接口或能力边界
- 技术澄清
- 架构决策
- 数据结构或字段变化
- API 合约
- 关键交互规则

不要写具体文件路径或代码片段，避免后续实现变化导致 PRD 过时。

## 测试决策

记录测试相关决策，包括：

- 什么样的测试是好测试：优先测试外部行为，而不是内部实现细节
- 哪些模块需要测试
- 代码库中是否存在可参考的相似测试
- 关键验收场景和异常场景

## 不在范围内

明确本次 PRD 不覆盖的内容、后续再做的能力，以及容易被误解为本期范围但实际不做的事项。

## 补充说明

记录其他重要上下文、风险、待确认事项、依赖关系或后续建议。

</prd-template>
